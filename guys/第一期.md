## 后端库表设计

### 标签表

建议用标签，不用分类，更灵活

- 性别（男女
- 方向（java、c++、前端
- 目标（考研、春招、秋招、考公、社招、竞赛（蓝桥杯、、科研
- 段位（初级、中级、高级、
- 身份（大一、大二、大三、大四、研一、研二、研三、就业
- 运动（篮球、足球、羽毛球、排球、

- 用户自定义标签

字段

- id int 主键 
- 标签名 varchar（必须唯一，唯一索引
- 上传用户 userId int（普通索引
- 是否为父标签 is_parent，tiny int（布尔类型不太灵活
- 父标签 id, parentId , int
- 创建时间 create_time， datetime
- 更新时间 update_time，datetime
- 是否删除 is_deleted，tiny int

如何查询所有标签，并且把标签分好组？
根据父标签查询子标签

```mysql
- ddl
create table tag
(
	id bigint auto_increment comment 'id' primary key,
  tag_name varchar(256) null comment '标签名称',
  userId bigint null comment '用户id',
  is_parent tinyint null comment '0-不是，1-父标签'
  parentId bigint null comment '父标签id',
  create_time datetime default CURRENT_TIMESTAMP null comment '创建时间',
  update_time datetime default CURRENT_TIMESTAMP null on update CURRENT_TIMESTAMP,
  is_deleted tinyint default 0 not null comment '是否删除'
  

)
```



### 用户表

如何存储用户和标签的关系？
根据自己的实际需求

1. 直接在用户表补充tags字段
    1. 查询方便，不用新建关联表，标签是用户的固有属性，节省开发成本(数据量不是很大)
    2. 根据标签寻找用户不方便，用户表多一列，会
    3. 如果要倒查，可以用like，如果查询变慢了，可以用redis缓存
2. 加一个关联表，记录用户和标签的关系
    1. 查询某个标签下有多少用户，查询灵活，可以正查反查
    2. 缺点：要多维护一个表，如果要查一个用户的所有标签会麻烦
    3. 企业大项目开发中尽量减少关联查询，而且会影响查询性能

```mysql
create table user
(
  username varchar(256) null comment '用户昵称',
)

alter table user add COLUMN tags varchar(1024) null comment '标签列表';
```
